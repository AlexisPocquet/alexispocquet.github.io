Research Experience

  • M1 Research Internship Université Paris-Saclay, LMF, Deducteam team · Thibaut Benjamin · march 2026 to july 2026

    Weak ω-categories are higher-dimensional algebraic structures generalizing the geometry of types in intentional type theory, and are a leading candidate for representing homotopy types — though many constructions within this representation remain poorly understood. The goal of this internship is to extend a type-theoretical presentation of weak ω-categories with parametricity axioms, corresponding to a class of cylinder construction operations.

  • L3 Research Work CNRS, ENS Paris, Antique team · Jérôme Féret and Jérôme Boillot · january 2025 to july 2025

    Over the course of my last bachelor semester and the following summer internship, I worked with the Antique team at ENS Paris under the supervision of Jérôme Féret and Jérôme Boillot, discovering Abstract Interpretation and its applications to system software formal verification. [more] Abstract Interpretation, introduced by Radhia and Patrick Cousot in the late 1970s, formalizes program analysis as computations over over-approximations of reachable program states, ensuring termination while retaining sufficient precision to infer meaningful invariants. My semester project focused on combining abstract domains through a construction known as a reduced product, studying several of its instantiations — such as those based on input/output channels or ghost quantities — and analyzing how they address termination and refinement of abstract computations.

    Building on this foundation, my internship extended this work towards operating system verification on the IA-32 architecture. I proposed a method to verify that critical OS properties are preserved during the execution of application tasks, by introducing an intermediate language with a formal semantics to model assembly instructions. From this, I derived a reachability semantics characterizing all states reachable during task execution, and constructed a sound and sufficiently precise over-approximation based on constraint-manipulating functions over memory states. I then defined a set of states proved to be stable under this abstract analysis, provided that application executions do not compromise the integrity of the host system's essential data structures.

Programming

  • KawaLang winter 2024

    Kawa is an object-oriented toy programming language developed as part of an introduction to the programming language theory course of Université Paris-Saclay. The implementation, in OCaml, includes a basic interpreter and a static type checker designed for exhaustiveness and precise type error reporting. [more] The interpreter is quite naive and the pipeline for code execution is divided in three sequential parts: code parsing and structures loading, static type type-checking and code evaluation/execution. These two last phases strongly depend on symbol/name resolution, which is permanently being used during the interpretating process.

    This language should not be used for real projects since its implementation is experimental and sometimes buggy. Here is a set of examples that I used to test my implementation.

  • TinyProver fall 2025

    TinyProver is an interactive proof assistant developed as part of the computational logic course at École Polytechnique. The implementation, in OCaml, provides two independent provers: one for propositional logic via simply typed lambda-calculus, and one for a minimal kernel of dependent type theory. [more] The simply typed prover supports type inference, bidirectional type checking, and an interactive tactic-based proof mode with sequent manipulation — including implication elimination, conjunction, disjunction, and the cut tactic. Proofs can be written and loaded from .proof files.

    The dependent types prover implements the core of Martin-Löf type theory: a substitution algorithm, parallel beta-reduction for normalization, alpha-conversion, and bidirectional type inference for Pi-types, natural numbers, and propositional equality. The normalization strategy proved to be the most delicate part, as a naive fixpoint approach was insufficient for terms defined by recursion on Nat. The project concludes with a reflection on the gap between a minimal kernel and a practically usable proof assistant.

  • MiniGoLang winter 2026

    MiniGoLang is a compiler for a subset of Go targeting the Intel x86-64 architecture, developed as part of the compilation course at École Polytechnique. The implementation, in OCaml, covers the full pipeline from source to assembly, with structured error accumulation throughout the front-end. [more] The front-end handles lexing, parsing, symbol resolution, and type checking. Errors are accumulated monadically via a dedicated Error module rather than failing at the first issue. The back-end produces x86-64 assembly using a stack-based compilation scheme, preceded by a rewriting pass that simplifies the typed AST before code generation.

    The compiler passes all provided test cases, which cover parsing rejection, typing correctness, and execution. Additional tests were written to validate nested struct access, forward function calls, and mutually recursive functions. Here is a set of test cases used during development, along with a detailed report of the implementation.

  • MiniRust spring 2025

    MiniRust is a school project exploring the internals of the Rust programming language. The implementation, in OCaml, includes a partial borrow checker and a backend that compiles a subset of Rust down to C code. It has been developped for the need of the advanced functional programming course of ENS Paris-Saclay. [more] The compilation pipeline runs a type checker and the borrow checker before emitting any C code. The borrow checker is incomplete and does not handle all edge cases, while the C backend produces non-idiomatic output relying heavily on labels — a direct consequence of a linear translation from the intermediate MiniMir representation. On 94 tests, the implementation passes 91 of them.

    The most technically involved part of the backend is the resolution of struct dependencies. It builds a dependency graph over structs, then performs a postfix DFS on each connected component starting from the least dependent node, ensuring definitions are emitted in the right order. Cyclic struct dependencies through indirection are handled implicitly; true cycles are left out of scope. Here is a set of test cases used to validate the implementation.
